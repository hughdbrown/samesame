  Code Review: samesame

  1. Is there a better way than O(n²) file comparison?

  Short answer: Not fundamentally, but there are optimizations unused here.

  The pairwise comparison of all files is inherently O(n²) for finding all duplicates. However, several optimizations could reduce the constant factor:

  - Size-based pre-filtering: Only compare files of similar size. A 10-line file and a 1000-line file can share at most 10 matching lines.
  - Full-file hash grouping: Compute a hash of the entire file content first; files with completely different hashes cannot share significant duplicates.
  - Min-hash / Locality-Sensitive Hashing: Could quickly estimate similarity and skip obviously dissimilar pairs.

  That said, for the stated use case (typically <10 files in a duplicate group), the current approach is appropriate.

  2. Is rayon usage optimal?

  Mostly good, with one concern:

  // main.rs:46-63 - File reading phase
  let files: Vec<FileDescription> = paths.par_iter().filter_map(...).collect();

  // main.rs:82-85 - Comparison phase  
  let results: Vec<ComparisonResult<'_>> = pairs.par_iter().map(...).collect();

  The parallelism is applied at the right granularity. However:

  - No work-stealing tuning: Rayon's defaults are generally good, but for I/O-bound file reading vs CPU-bound diffing, different thread pool configurations might be optimal.
  - Barrier between phases: All files must be read before any comparisons begin. For very large sets, a pipeline approach could start comparing as files become available.

  Top 5 Criticisms

  1. LCS Algorithm Has O(n²) Time and Space Complexity (diff.rs:338-378)

  pub fn compute_lcs(hashes1: &[u64], hashes2: &[u64]) -> Vec<(usize, usize)> {
      // ...
      let mut dp = vec![vec![0usize; n + 1]; m + 1];  // O(m*n) memory!

  For two 5000-line files, this allocates ~200MB just for the DP table. The patience diff mitigates this by using LCS only for sections without unique anchors, but degenerate cases (files with many repeated lines like test fixtures or generated code) will hit this.

  Fix: Use Hirschberg's algorithm for O(n) space LCS, or Hunt-Szymanski for sparse matches.

  2. Line Content Stored Eagerly for All Files (file.rs:58-72)

  for line in text.lines() {
      lines.push(line.to_string());  // Allocated for EVERY file
      hashes.push(hash_line(line.trim()));
  }

  The lines field is only used during output formatting for files that have matches. For a 100-file comparison where only 5 pairs have duplicates, 95 files have their content stored uselessly.

  Fix: Store only hashes during comparison. Re-read file content on demand in format_text/format_json, or use memory-mapped I/O.

  3. LIS Algorithm is O(n²) When O(n log n) is Achievable (diff.rs:245-283)

              if matches[j].1 < matches[i].1 && dp[j] + 1 > dp[i] {

  The classic O(n log n) LIS algorithm using binary search and patience sorting is well-known and would be a significant improvement for files with many unique lines.

  4. No Integration Tests for the Binary

  The test suite thoroughly covers individual modules but has no tests that:
  - Execute the actual main() function
  - Test CLI argument parsing edge cases (e.g., --match 0, conflicting flags)
  - Verify exit codes under different conditions
  - Test the full pipeline with real files on disk

  This gap means regressions in the orchestration layer (main.rs) could go undetected.

  5. Regex Filter Applied at Output Time, Not Comparison Time (output.rs:61-72, 106-108)

  The regex filter is applied during output formatting. This means:
  1. All file pairs are compared even if most matches will be filtered
  2. All ComparisonResult objects are created and stored
  3. Wasted CPU and memory for matches that are never displayed

  Fix: Apply the regex filter during or immediately after compare_files, before collecting results.

  Algorithmic Question #17 from CLAUDE.md

  Regarding "Duplicate Processing": The code correctly generates only (i, j) pairs where i < j:

  // discovery.rs:118-120
  for i in 0..count {
      for j in (i + 1)..count {
          pairs.push((i, j));
      }
  }

  This avoids comparing both (A, B) and (B, A). The patience diff is symmetric—comparing A to B yields the same matches as B to A, just with ranges swapped. The current implementation is correct and efficient.

  Test Strategy for Filesystem-Dependent Code

  For the untested filesystem operations, I recommend:

  1. Transient test files (already used): The tempfile crate creates real files that auto-cleanup. This is appropriate for read_file, is_binary_file, etc.
  2. Property-based testing: Use proptest or quickcheck to generate random file contents and verify diff properties (e.g., identical files → one Same range covering all lines).
  3. Snapshot testing: For output formatting, use insta to capture expected outputs and detect regressions.
  4. assert_cmd for CLI testing: Test the actual binary with different argument combinations and verify exit codes and output patterns.
