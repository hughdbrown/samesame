  Repository Quality Review: samesame

  Overview

  A well-structured Rust CLI tool for finding duplicate code using rolling XOR
  hashes. Clean modular architecture, good test coverage (110 tests), and solid
  error handling with thiserror. The recent rolling hash refactor is
  well-executed.

  Per CLAUDE.md item 9, version should be 1.1.1, currently 0.2.0.

  2. Unused registry parameter in filter_groups_by_regex (main.rs:35-56)
  The function takes registry: &FileRegistry but never uses it вҖ” suppressed with
   let _ = (registry, path); which is a code smell. Remove the parameter
  entirely.

  read_file_if_text calls is_binary_file (reads first 8KB), then read_file
  (reads the entire file again). This reads the file from disk twice. Combine

      }

  }

  ---
  Medium Priority

  5. Public API surface is too wide in rolling_hash.rs
  compute_rolling_hashes, group_blocks, blocks_to_duplicate_groups, and
  BlockDescriptor are all pub but only used internally. Only find_duplicates,
  FileRegistry, and DuplicateGroup need to be public. Use pub(crate) or make
  them private.

  6. Public #[doc(hidden)] functions should be pub(crate) instead
  In file.rs: hash_line. In discovery.rs: scan_glob, is_symlink, is_hidden,
  deduplicate_paths. The #[doc(hidden)] pattern is meant for re-exports; for
  internal-but-testable functions, pub(crate) is idiomatic.

  7. String formatting allocations in output.rs
  Multiple format!() + push_str() calls allocate intermediate strings. Use
  write!() instead:
  use std::fmt::Write;
  write!(output, "  {}  lines {}-{}\n", path.display(), start + 1,
  end).unwrap();

  8. serde_json error silently swallowed (output.rs:131)
  serde_json::to_string_pretty(&output).unwrap_or_else(|_| "{}".to_string())
  Serialization of known types shouldn't fail, but if it does, returning {}
  hides the bug. Use .expect("serialization of known types") or propagate the
  error.

  9. FileDescription.len() has #[allow(dead_code)] (types.rs:18)
  This method is only used in tests. Either remove the allow and accept the
  warning, or use #[cfg(test)] if it's truly test-only. Since it's a natural
  part of the API, just remove the #[allow(dead_code)] вҖ” it's used in
  integration tests so the warning won't fire.

  ---
  Low Priority

  10. find function defined inside consolidate_regions (rolling_hash.rs:370)
  The recursive union-find find with path compression is defined as a nested fn.
   This works but is unusual. Consider extracting it or using an iterative
  version to avoid potential stack overflow on pathological inputs (thousands of
   chained regions).

  11. XOR hash is order-independent (known trade-off)
  Lines {A, B, C, D, E} and {E, D, C, B, A} produce the same rolling hash,
  yielding false positives. This was accepted by the user. Worth a comment in
  the module doc noting the false-positive-but-no-false-negative property.

  12. keyword in Cargo.toml still says "diff"
  keywords = ["duplicate", "code", "diff", "analysis"]
  The project no longer uses diff вҖ” consider replacing with "hash" or
  "detection".

  13. CLAUDE.md references deleted modules
  The Architecture section still documents diff.rs, grouping.rs, union_find.rs,
  Range, LineRange, ComparisonResult, and the patience diff algorithm. Should be
   updated to reflect the rolling hash architecture.

  14. file.rs:62 вҖ” unnecessary to_string() allocation per line
  lines.push(line.to_string());
  Since text is already a Cow<str>, and lines() returns &str slices, the
  .to_string() is necessary. No fix needed вҖ” just noting it's inherent.

  15. No parallelism in the rolling hash pipeline
  File reading is parallelized with rayon, but find_duplicates runs
  single-threaded. For very large codebases, compute_rolling_hashes could be
  parallelized per-file. Low priority since the hash computation is fast
  relative to I/O.
