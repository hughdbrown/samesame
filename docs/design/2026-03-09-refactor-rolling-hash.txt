❯ Will the non-overlapping in the same file be enough on its own? Can we remove the matching first and last hash rule we added?

⏺ They address different problems — I'd recommend keeping both.

  The overlap filter only catches same-file matches where ranges overlap. It does nothing for:

  1. Cross-file false positives — Two different files with brace-heavy sections could have the same XOR hash despite different actual
   content (since XOR is order-independent). The first/last hash check eliminates many of these.
  2. Same-file non-overlapping false positives — Two distant regions in the same file could have identical XOR hashes from different
  line arrangements. The overlap filter wouldn't catch these since the ranges don't overlap, but first/last hash would.

  The composite key improves hash discrimination in general. The overlap filter is a structural rule about what constitutes a valid
  self-match. They're complementary.
